# QA Test Scenarios — File List, Sharing & Preview

Covers the changes from the file-list session: duplicate-PDF fix, connector resolution refactor, preview auth hardening, and Shares tab Preview icon.

---

## 1. Duplicate PDF Prevention (firma_managed tag)

**Context:** Watermark toggle re-generates a PDF and uploads it to Google Drive. Previously these generated PDFs appeared in the file browser. Fix: `listFiles` now excludes files tagged `firma_managed: 'true'`.

| # | Scenario | Steps | Expected |
|---|----------|-------|----------|
| 1.1 | Generated PDF hidden from file browser | Grant share with watermark ON → toggle watermark OFF | Only the original DOCX visible in Files tab; no generated PDF(s) visible |
| 1.2 | Toggle does not accumulate PDFs | Toggle watermark ON → OFF → ON → OFF 3 times | Still only one PDF in Drive (overwrite via `sharedPdfDriveId`); file browser shows zero PDFs |
| 1.3 | Existing firma_managed files | Files with `appProperties.firma_managed = 'true'` pre-existing in Drive folder | They do not appear in file browser and are not indexed by Inngest |
| 1.4 | Normal files unaffected | Upload a regular PDF (not generated by app) | Visible in file browser and indexed normally |

## 2. Download & Preview Disposition (P0 fix)

**Context:** Cached PDF fast-path in `download-share/route.ts` was sending `Content-Disposition: attachment` even for preview requests, causing forced downloads instead of inline rendering.

| # | Scenario | Steps | Expected |
|---|----------|-------|----------|
| 2.1 | Preview renders inline | Share a DOCX with watermark; open Preview from Files tab | PDF renders in in-app iframe; browser does not prompt download |
| 2.2 | Download still works | Share same DOCX; click Download | File downloads as attachment with correct filename |
| 2.3 | Non-cached preview | Clear app cache / first request for a file | Preview still renders inline (non-cached path also sets `inline`) |
| 2.4 | Preview filename preserved | Inspect response headers for a cached preview | `Content-Disposition: inline` (no filename in preview path) |

## 3. Connector Resolution (`resolveEngagementConnectorId`)

**Context:** Connector lookup logic (document → client → firm fallback chain) was duplicated across 5 routes. Now centralised in `resolveEngagementConnectorId`.

| # | Scenario | Steps | Expected |
|---|----------|-------|----------|
| 3.1 | Document-level connector | Document has its own `connectorId` set | Routes use that connector directly |
| 3.2 | Client-level fallback | Document has no connector; client has connector | Routes use the client connector |
| 3.3 | Firm-level fallback | Neither document nor client have connectors; firm has one | Routes use the firm connector |
| 3.4 | No connector at all | Firm, client, and document all have no connector | Routes return appropriate error (400/404/500 depending on route) |
| 3.5 | Routes affected | Test each route independently: preview, download-share, bulk-download, search, index-file-intake | All behave correctly in each fallback scenario above |

## 4. Preview Auth — EV/EC Permission Check (hardening)

**Context:** `preview/route.ts` used to check `engagementDocument.findFirst` for EV/EC access (confirms indexing, not sharing intent). Now checks `engagementDocumentSharingUser` with `sharingPermissionStatus: 'GRANTED'`.

| # | Scenario | Steps | Expected |
|---|----------|-------|----------|
| 4.1 | EV can preview shared doc | Share doc with EV with status GRANTED; EV opens Preview | Preview renders (200) |
| 4.2 | EV blocked on unshared doc | Doc indexed but not shared with EV; EV hits preview URL directly | 403 Access denied |
| 4.3 | EC can preview shared doc | Share doc with EC; EC opens Preview from Shares tab | Preview renders (200) |
| 4.4 | EC blocked after revoke | Grant share → revoke → EC hits preview URL | 403 Access denied |
| 4.5 | EL/EM unaffected | EL or EM (internal role) opens any indexed document preview | Preview renders (200) — sharing check bypassed for internal roles |
| 4.6 | PENDING share blocked | `sharingPermissionStatus` is `PENDING` not `GRANTED` | 403 Access denied |

## 6. Folder Session Persistence (EL vs EV)

**Context:** EL folder navigation is persisted in `sessionStorage` keyed by project. EV should not persist (no folder context).

| # | Scenario | Steps | Expected |
|---|----------|-------|----------|
| 6.1 | EL folder state remembered | EL navigates into a subfolder; refreshes page | Returns to same folder |
| 6.2 | EV starts at root | EV opens Files tab | Always starts at root; no remembered folder |
| 6.3 | Different projects isolated | EL navigates to subfolder in Project A; switches to Project B | Project B starts at its own last-visited folder, not Project A's |
